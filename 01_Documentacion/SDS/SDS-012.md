# 2. Objetivo

Implementar el proceso de inicialización de la Base Vectorial e integrar completamente el pipeline RAG del proyecto Mercado Central AI, validando el funcionamiento de extremo a extremo mediante una interfaz desarrollada con Streamlit.

El Sprint 14 representa el cierre funcional del proyecto, consolidando todos los módulos desarrollados durante los Sprint anteriores en una solución RAG completamente operativa.

---

# 3. Contexto Arquitectónico

Al finalizar el Sprint 13, el proyecto contaba con una interfaz funcional desarrollada en Streamlit y con todos los componentes principales del pipeline implementados.

Durante el Sprint 14 se incorporó el proceso de inicialización de la Base Vectorial, permitiendo cargar automáticamente la Base de Conocimiento, generar embeddings, construir la colección vectorial y dejar disponible el sistema para responder consultas utilizando recuperación semántica.

Asimismo, se validó la integración completa entre todos los módulos del proyecto, asegurando el funcionamiento del flujo conversacional desde la interfaz de usuario hasta la generación de respuestas mediante Google Gemini.

La arquitectura final es la siguiente:

```text
Usuario
      │
      ▼
Streamlit
      │
      ▼
RAG Pipeline
      │
      ▼
Retriever
      │
      ▼
Context Builder
      │
      ▼
Decision Engine
      │
      ▼
Prompt Builder
      │
      ▼
Google Gemini Provider
      │
      ▼
Gemini 3 Flash Preview
      │
      ▼
Respuesta
```

La Base Vectorial se inicializa previamente mediante el proceso de carga documental, división en fragmentos, generación de metadata, creación de embeddings y almacenamiento persistente en ChromaDB.

# 6. Arquitectura del módulo

La integración final mantiene la arquitectura basada en Interfaces, Factory Pattern y Principio de Responsabilidad Única (SRP).

La organización principal del proyecto es la siguiente:

```text
src/
│
├── config/
├── knowledge/
├── embeddings/
├── vector_store/
├── retriever/
├── context/
├── decision_engine/
├── prompt/
├── llm/
├── rag/
├── ui/
└── tools/
```

Cada módulo mantiene responsabilidades claramente definidas y puede evolucionar independientemente del resto del sistema.

La orquestación del flujo completo queda centralizada en:

```text
RAGPipeline
```

mientras que la inicialización de la Base Vectorial es responsabilidad exclusiva de:

```text
KnowledgeInitializer
```

Esta separación evita acoplamientos innecesarios entre el proceso de indexación y el proceso conversacional.

# 7. Componentes del módulo

## 7.1 KnowledgeInitializer

Responsable de construir la Base Vectorial del proyecto.

Funciones principales:

- cargar documentos;
- dividir documentos;
- procesar metadata;
- generar embeddings;
- almacenar documentos en ChromaDB;
- verificar la colección creada.

---

## 7.2 RAGPipeline

Responsable de coordinar el flujo conversacional completo.

Funciones principales:

- recibir consultas;
- recuperar documentos relevantes;
- construir el contexto;
- construir el prompt;
- solicitar la respuesta al proveedor LLM.

---

## 7.3 Retriever

Responsable de recuperar los documentos más relevantes utilizando búsqueda semántica sobre ChromaDB.

---

## 7.4 Context Builder

Responsable de consolidar los fragmentos recuperados en un único contexto que será utilizado por el modelo de lenguaje.

---

## 7.5 Decision Engine

Responsable de preparar la solicitud que será utilizada para construir el prompt definitivo.

---

## 7.6 Prompt Builder

Responsable de generar el prompt final enviado al proveedor LLM.

---

## 7.7 Google Gemini Provider

Responsable de encapsular completamente la comunicación con Google Gemini mediante LangChain.

---

## 7.8 StreamlitApp

Punto de entrada de la aplicación.

Responsabilidades:

- recibir consultas del usuario;
- invocar el pipeline RAG;
- presentar la respuesta generada.

# 10. Patrón arquitectónico

La solución implementa los siguientes patrones de diseño:

| Patrón | Aplicación |
|---------|------------|
| Interface | Contratos públicos entre módulos. |
| Factory Pattern | Creación centralizada de proveedores y componentes. |
| Dependency Inversion | Los consumidores dependen de interfaces y no de implementaciones concretas. |
| SRP | Cada módulo mantiene una única responsabilidad claramente definida. |
| OCP | La arquitectura permite incorporar nuevos proveedores y componentes sin modificar el código existente. |
| Modular Architecture | Separación funcional por dominios de responsabilidad. |

La integración realizada durante el Sprint 14 preserva completamente estos principios arquitectónicos, consolidando un pipeline RAG desacoplado, mantenible y preparado para futuras extensiones.

# 14. Contrato de la API (API Contract)

## Precondiciones

Antes de procesar una consulta deberán cumplirse las siguientes condiciones:

- La Base Vectorial debe encontrarse inicializada.
- La colección ChromaDB debe existir.
- El proveedor de embeddings debe estar configurado.
- El proveedor LLM debe encontrarse correctamente inicializado.
- La API Key de Google Gemini debe ser válida.
- El pipeline RAG debe haber sido construido mediante su Factory correspondiente.

## Postcondiciones

Después de ejecutar el pipeline se garantiza que:

- se recuperan documentos relevantes mediante búsqueda semántica;
- se construye un contexto consolidado;
- se genera un prompt consistente;
- se obtiene una respuesta textual (`str`);
- la respuesta utiliza únicamente información proveniente de la Base de Conocimiento.

| ID | Regla |

Estas reglas garantizan la uniformidad arquitectónica y preservan el desacoplamiento definido durante el proyecto.

# 17. Consideraciones de rendimiento

La arquitectura fue diseñada para minimizar la sobrecarga durante el procesamiento de consultas.

Para ello se adoptaron las siguientes estrategias:

- inicialización única de la Base Vectorial;
- reutilización de la colección persistente de ChromaDB;
- recuperación semántica limitada mediante `top-k`;
- construcción incremental del contexto;
- reutilización del cliente Google Gemini;
- inicialización única del pipeline mediante `@st.cache_resource` en Streamlit.

Estas decisiones reducen significativamente el tiempo de respuesta y evitan operaciones repetitivas durante la ejecución de la aplicación.

La arquitectura permanece preparada para incorporar futuras optimizaciones sin modificar la interfaz pública de los componentes.

# 20. Registro de Decisiones Arquitectónicas (ADR)

| ADR | Decisión | Justificación |
|-----|----------|---------------|
| ADR-012-01 | Mantener el pipeline completamente desacoplado. | Facilita el mantenimiento y la evolución del sistema. |
| ADR-012-02 | Centralizar la orquestación en `RAGPipeline`. | Evita duplicación de lógica entre consumidores. |
| ADR-012-03 | Separar el proceso de inicialización de la Base Vectorial del flujo conversacional. | Permite reutilizar la Base Vectorial sin reconstruirla en cada ejecución. |
| ADR-012-04 | Mantener ChromaDB como almacén vectorial persistente. | Reduce tiempos de inicialización y mejora el rendimiento. |
| ADR-012-05 | Utilizar Streamlit como interfaz oficial del proyecto. | Simplifica la demostración y validación funcional. |
| ADR-012-06 | Encapsular Google Gemini mediante `GoogleGeminiProvider`. | Evita dependencias directas con LangChain y facilita futuras migraciones de proveedor. |
| ADR-012-07 | Mantener la configuración centralizada mediante `settings.py`. | Favorece la mantenibilidad y evita valores hardcodeados. |
| ADR-012-08 | Validar el sistema mediante pruebas End-to-End. | Garantiza el correcto funcionamiento del pipeline completo antes de la publicación de la Release Final. |

# 21. Riesgos identificados

| Riesgo | Mitigación |
|---------|------------|
| Cambios en la API de Google Gemini. | Encapsular la integración dentro de `GoogleGeminiProvider`. |
| Cambios en LangChain. | Limitar el uso de LangChain únicamente al Provider. |
| Crecimiento de la Base de Conocimiento. | Utilizar ChromaDB como almacén vectorial persistente y escalable. |
| Cambios en la configuración. | Centralizar todos los parámetros mediante `settings.py`. |
| Incorporación de nuevos proveedores LLM. | Mantener el uso de Interfaces y Factory Pattern. |
| Evolución futura del sistema. | Preservar la arquitectura modular implementada durante el proyecto. |
